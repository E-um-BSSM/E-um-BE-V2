# 개발 컨벤션

이 문서는 이 저장소에서 커밋 · 브랜치 · 이슈 · PR을 작성하는 규칙을 정의합니다.

## 커밋 컨벤션

| 유형 | 설명 |
|---|---|
| feat | 새로운 기능 추가 |
| fix | 버그 수정 |
| refactor | 코드 리팩토링 (기능 변경 없이 구조 개선) |
| test | 테스트 코드 작성 |
| chore | 기타 자잘한 작업 (빌드 설정, 패키지 관리 등) |
| docs | 문서 추가 또는 수정 |
| delete | 불필요한 코드나 파일 삭제 |
| build | 빌드 관련 파일 및 설정 변경 |

커밋 메시지 형식: `유형: 변경 사항 요약`
- 제목은 30자 이내, 본문은 자유롭게
- 예시: `feat: 클래스 API 구현`

## 브랜치 컨벤션

형식: `유형/#이슈번호` (예: `feat/#1`)

- 하나의 이슈 = 하나의 브랜치 = 하나의 PR을 기본 단위로 합니다.
- 브랜치는 최신 `main`에서 분기합니다.

## 이슈 작성 규칙

- 제목: `[유형] 작업 요약` (예: `[feat] 클래스 API 구현`)
- 본문 템플릿:

  ```md
  ## 작업 내용
  - 무엇을 / 왜 하는지 설명

  ## 세부 작업
  - [ ] 하위 작업 1
  - [ ] 하위 작업 2

  ## 참고
  - 관련 API 명세 / ERD / 문서 링크
  ```

- 라벨: 커밋 유형과 동일한 이름의 라벨을 부착합니다 (feat / fix / refactor / test / chore / docs / delete / build).

## PR 작성 규칙

- 브랜치: `유형/#이슈번호` → `main`
- 제목: 커밋 메시지와 동일한 형식 `유형: 변경 사항 요약`
- 본문 템플릿:

  ```md
  ## 관련 이슈
  Closes #이슈번호

  ## 변경 사항
  - 주요 변경점을 bullet로 정리

  ## 테스트
  - 확인한 내용 / 확인하지 못한 사항
  ```

- 머지 방식: 일반 Merge commit 사용 (Squash 금지 — 이슈별 커밋 히스토리를 보존하기 위함).
- 리뷰어가 없는 개인 작업이라도 PR을 통해 머지하며, `Closes #이슈번호`로 이슈를 자동 종료합니다.
